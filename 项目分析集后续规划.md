# 二、评分要求重新分析：这个比赛不是预测比赛，是套利比赛

你最终提交文件包含：

```text
times,实时价格,power
```

但真正决定分数的是 `power`。官方评分是：

[
Profit_d=\sum_{t=0}^{95} P_{d,t}^{true} \cdot E_{d,t}
]

最终：

[
Score=\frac{1}{D}\sum_{d=1}^{D}Profit_d
]

也就是说：

* 你提交的预测电价本身不是最终目标；
* 预测电价只是为了选充电窗口和放电窗口；
* 关键是每天能不能选中“低价连续 2 小时充电”和“高价连续 2 小时放电”。

你的优化器 `optimize_one_day` 的思路是对的：它对一天 96 点价格计算 8 点窗口和，然后枚举充电开始点和放电开始点，找最大预测价差。代码里先计算 `block_sum`，然后枚举 `tc` 和 `td`，用 `power_value * (block_sum[td] - charge_cost)` 作为收益，超过阈值才交易。

所以，项目的核心不是：

> 让每个点预测得更准。

而是：

> 让最高 8 点放电窗口和最低 8 点充电窗口的相对排序更准。

这会导致一个重要结论：

**RMSE 降低，不一定分数提高；窗口排序提升，才会分数提高。**

# 三、为什么你做了很多改进但分数没涨？

## 1. 训练目标仍然是 RMSE，不直接对齐收益

`train_lgb.py` 中 LightGBM 默认参数是：

```python
"objective": "regression",
"metric": "rmse"
```

这说明模型训练和 early stopping 仍然围绕点预测误差展开。

但比赛评分不是 RMSE，而是收益。于是会出现：

| 情况                    |      RMSE |   比赛收益 |
| --------------------- | --------: | -----: |
| 整体曲线偏移，但高低窗口排序正确      |      可能较差 |   可能很好 |
| 价格大部分点预测准确，但最高/最低窗口错位 |      可能很好 |   可能很差 |
| 尖峰预测错 1 小时            | RMSE 小幅变化 | 收益大幅下降 |

所以你现在的问题很可能是：**模型在点预测上变强了，但在“选窗口”上没有变强。**

---

## 2. 你已经有收益阈值搜索，但只是后处理，不是训练目标

`validate_profit.py` 会对不同 threshold 做回测，统计平均收益、交易天数、亏损天数等。

这是正确方向，但还不够。现在流程是：

1. 先训练 RMSE 模型；
2. 用预测结果生成策略；
3. 搜阈值。

这相当于“先预测，再修补策略”。

更强的方案应该是：

1. 直接评估每天所有候选充放电窗口；
2. 训练模型学习哪个窗口组合收益最高；
3. 用收益、后悔值、亏损率作为模型选择指标。

也就是从 **point forecasting** 转成 **decision-focused forecasting**。

## 4. NWP 气象特征可能存在严重时间/维度错位

你赛题说明里给的 `.nc` 维度是：

```text
time, channel, hour, lat, lon
```

但你代码里写的是：

```python
# Official shape: time x hour x channel x lat x lon.
data = data[0]
for hour in range(data.shape[0]):
    arr = data[hour, channels.index(var), :, :]
```



如果官方文件真实维度确实是：

```text
time x channel x hour x lat x lon
```

那么你现在的索引就错了。你把 `channel` 当成了 `hour`，把 `hour` 当成了 `channel`。这样构造出的 NWP 特征会是错乱的，模型加了气象数据后反而可能下降。

另一个问题是：赛题说明 `time` 是世界时 UTC，而边界条件和电价是北京时间。你当前代码直接用：

```python
publish_date = pd.Timestamp(nc_path.stem)
target_date = publish_date + timedelta(days=1)
row = {TIME_COL: target_date + pd.Timedelta(hours=hour)}
```

也就是忽略了 `.nc` 文件中的真实 `time` 变量和 UTC 到北京时间的转换。

这很危险。NWP 一旦整体错 8 小时，光伏 GHI、云量、温度日变化都会错位，模型会学到错误关系。

**这是当前最值得优先排查的 bug。**

## 5. NWP 只做全区域 mean/std/min/max，空间信息太粗

`nwp_features.py` 目前对每个变量做的是：

* mean
* std
* min
* max

并且对整个网格直接聚合。

这对电价预测可能太粗。蒙西节点电价和风电/光伏出力不是由全区域平均气象决定的，而是由关键新能源场站区域、输电约束区域、负荷区域共同决定。

更好的做法是：

1. 不只做全图 mean；
2. 按经纬度分块；
3. 取西部、东部、北部、南部区域统计；
4. 对风速、GHI 做分位数特征；
5. 做 `wind_speed = sqrt(u100^2 + v100^2)`；
6. 做 `wind_speed^2`、`wind_speed^3`，因为风电功率和风速强非线性；
7. 对 GHI 加白天 mask；
8. 对云量和 GHI 做交互项。

当前 NWP 特征的“有”和“有效”不是一回事。你已经接入了，但很可能没有正确释放价值。

## 7. 联络线方向可能被默认假设了

你在 `features.py` 中构造了：

```python
net_load_with_intertie =
    load - wind - pv - hydro - non_market - intertie
```



这个写法隐含假设：联络线预测值为正时，相当于外部送入本区，减少本地净负荷。

但如果数据定义中正值表示本区外送，那么这个符号就反了。

这个不能靠猜，要用历史数据验证：

```python
corr(price, intertie)
corr(price, load - wind - pv - hydro - non_market - intertie)
corr(price, load - wind - pv - hydro - non_market + intertie)
```

哪个相关性更符合电价机理，就用哪个。建议同时保留两个版本：

```text
net_load_intertie_minus
net_load_intertie_plus
```

让模型自己选择。

现在你已经统计了 `avg_profit`、`traded_days`、`loss_days`。

| 指标                     | 公式/含义            |
| ---------------------- | ---------------- |

| `regret`               | 事后最优收益 - 你的收益    |

| `window_rank_corr`     | 候选窗口收益排序相关性      |

核心不是看 MAE，而是看：

```text
capture_ratio ↑
loss_days ↓
avg_profit ↑
regret ↓
```

* 月度 slot 统计；
* 模型递归预测值，但要谨慎。

---

## 2. 增加日级供需形态特征

不要只给模型每个 15 分钟点的局部特征，还要给它当天整体形态。

例如对每天的预测边界条件计算：

```text
daily_load_max
daily_load_min
daily_load_mean
daily_load_peak_valley
daily_net_load_max
daily_net_load_min
daily_net_load_peak_valley
daily_renewable_ratio_mean
daily_pv_peak
daily_wind_mean
daily_wind_ramp_max
daily_intertie_mean
daily_non_market_mean
```

这些特征可以帮助模型判断：

* 今天是否整体供给宽松；
* 今天是否容易低价；
* 今天高价可能出现在早高峰、晚高峰还是全天都不高。

---

## 3. 增加候选窗口特征

因为最终只选 8 点窗口，所以可以直接构造窗口级特征。

对每个可能充电窗口 (tc)：

```text
charge_window_price_prior_sum
charge_window_net_load_mean
charge_window_renewable_ratio_mean
charge_window_pv_mean
charge_window_wind_mean
charge_window_load_mean
```

对每个可能放电窗口 (td)：

```text
discharge_window_net_load_mean
discharge_window_load_mean
discharge_window_renewable_ratio_mean
```

对每组候选 `(tc, td)`：

```text
window_gap = td - tc
net_load_spread = discharge_net_load_mean - charge_net_load_mean
renewable_spread = charge_renewable_ratio - discharge_renewable_ratio
prior_price_spread = discharge_prior_price_sum - charge_prior_price_sum
```

这会为后面的 ranking 模型打基础。

---

## 4. NWP 特征升级

当前 NWP 是全网格 mean/std/min/max。建议升级为：

### 风电相关

```text
wind_speed_mean
wind_speed_std
wind_speed_p10
wind_speed_p50
wind_speed_p90
wind_speed_max
wind_speed_cube_mean
u100_mean
v100_mean
wind_direction_sin
wind_direction_cos
```

风电和风速不是线性关系，尤其在切入风速、额定风速附近，非线性很强。因此 `wind_speed^2`、`wind_speed^3` 很有价值。

### 光伏相关

```text
ghi_mean
ghi_max
ghi_p90
tcc_mean
ghi_x_1_minus_tcc
daylight_ghi_sum
noon_ghi_mean
```

GHI 应该主要影响白天。建议夜间 GHI 直接置零或加 daylight mask。

### 温度负荷相关

```text
t2m_celsius_mean
t2m_celsius_max
t2m_celsius_min
heating_degree
cooling_degree
```

冬季蒙西负荷和温度可能有关系，可以构造：

```python
heating_degree = max(0, 18 - t2m_celsius)
```

不要引入外部气温舒适阈值数据，只是规则特征。

# 九、第四阶段：模型路线

## 1. 保留 LightGBM，但要换模型目标组合

现在 LightGBM 是合理主模型，不建议直接推翻。你可以训练多套：

| 模型               | 目标      | 用途   |
| ---------------- | ------- | ---- |
| LGB-RMSE         | 普通价格预测  | 基础曲线 |
| LGB-L1           | 抗异常点    | 稳健预测 |
| LGB-Huber        | 尖峰不过拟合  | 稳健   |
| LGB-Quantile 0.2 | 低价保守估计  | 充电窗口 |
| LGB-Quantile 0.8 | 高价保守估计  | 放电窗口 |
| CatBoost         | 表格非线性补充 | 集成   |
| XGBoost          | 补充树模型差异 | 集成   |

策略层可以这样用：

```text
充电窗口：参考 q20 / mean
放电窗口：参考 q80 / mean
交易触发：q80_discharge - q20_charge > threshold
```

这样比单点预测更稳。

---

## 2. 做残差模型，而不是直接预测原始电价

节点电价强季节性、强日内性。建议改为：

[
P_{d,t}=SeasonalPrior_{d,t}+Residual_{d,t}
]

第一步：用历史构造季节先验：

```text
month_slot_mean
dow_slot_mean
same_day_last_year
recent_same_slot_mean
```

第二步：LightGBM 预测残差：

```text
residual = true_price - seasonal_prior
```

最后：

```text
pred_price = seasonal_prior + pred_residual
```

你的 `make_blended_submission.py` 已经有类似思想：它把模型价格和月度 slot 先验按 alpha 混合。

但我建议把它前移到训练阶段，而不是只在提交阶段混合。

---

## 3. 加一个窗口排序模型

这是最重要的提分方向。

你不要只预测 96 个点价格，而是每天生成所有合法候选：

[
(tc,td), \quad 0\le tc\le80,\quad tc+8\le td\le88
]

每个候选的真实标签是：

[
y_{tc,td}=1000\left(\sum_{td}^{td+7}P_t-\sum_{tc}^{tc+7}P_t\right)
]

然后训练一个模型直接预测候选窗口收益。

特征包括：

```text
tc
td
td - tc
charge_slot
discharge_slot
charge_prior_sum
discharge_prior_sum
predicted_net_load_spread
renewable_ratio_spread
load_spread
pv_spread
wind_spread
nwp_spread
```

模型可以用：

* LightGBM regression；
* LightGBM ranker；
* XGBoost ranker；
* CatBoost ranking。

最终每天选择预测收益最高的 `(tc, td)`。

这比单点价格预测更贴近评分。

---

## 4. 加一个“是否交易”分类模型

有些天真实峰谷价差小，交易容易亏。赛题允许不交易，收益为 0。

建议训练二分类模型：

```text
label = 1 if oracle_best_profit > 某阈值 else 0
```

输入：

```text
daily features
predicted spread
daily net load spread
renewable ratio
price prior spread
model uncertainty
```

最后策略：

```python
if trade_model_prob < p_threshold:
    power = 0
else:
    use best window
```

这通常能显著降低亏损天数。

---

# 十、策略层优化

## 1. 不要只有一个全局 threshold

现在阈值网格是：

```text
0,5000,10000,20000,30000,50000,80000,100000
```

README 和代码里都有这个逻辑。

但不同月份、不同市场状态下，阈值应不同。

建议改成：

```text
threshold_by_month
threshold_by_trade_confidence
threshold_by_predicted_volatility
threshold_by_spread_quantile
```

例如：

```python
if month == 1:
    threshold = threshold_jan
elif month == 2:
    threshold = threshold_feb
```

或者：

```python
risk_adjusted_spread = pred_spread - lambda_ * spread_error_std
trade if risk_adjusted_spread > 0
```

---

## 2. 用稳健收益而不是预测收益

当前优化器最大化：

```text
pred_discharge_sum - pred_charge_sum
```

可以改成：

```text
robust_score =
    pred_spread
    - λ * historical_error_std_of_this_window
    - μ * model_ensemble_disagreement
```

如果多 seed 模型对某天窗口选择分歧很大，就降低交易概率。

---

## 3. 多策略集成，而不是只提交一套逻辑

你可以本地生成多套候选：

| 策略                  | 说明              |
| ------------------- | --------------- |
| model_unconstrained | 纯模型，全窗口枚举       |
| model_constrained   | 限制经验窗口          |
| seasonal_only       | 月度先验窗口          |
| blend_0.2           | 20% 模型 + 80% 季节 |
| blend_0.4           | 40% 模型 + 60% 季节 |
| quantile_safe       | q80-q20 保守价差    |
| ranker_strategy     | 窗口排序模型          |

然后在验证集上比较：

```text
avg_profit
loss_days
capture_ratio
trade_rate
```

最终不要选 MAE 最低的，而选验证收益最稳的。

# 十一、前沿论文方法如何借鉴

注意：不能使用外部数据，但可以借鉴模型结构和思想。

可以参考这些方向：

## 1. TFT / Temporal Fusion Transformer

适合多变量时间序列，能处理已知未来变量，比如负荷预测、新能源预测、气象预测。

适合你的场景：

```text
已知未来边界条件 + 历史价格 + 时间特征 -> 预测未来价格
```

但缺点是数据量不够时容易不如 LightGBM。

---

## 2. PatchTST

PatchTST 对长序列时间序列预测效果强，适合把一天 96 点作为 patch 或把多日序列输入。

用法：

```text
输入过去 N 天价格 + 未来边界条件
输出未来 96 点价格
```

建议作为集成模型，不建议一开始替代 GBDT。

---

## 3. TimesNet

TimesNet 对多周期时间序列有优势。电价具有日周期、周周期、季节周期，可以尝试。

---

## 4. N-BEATS / N-HiTS

适合做强时间序列基线，尤其是趋势/季节分解。

你的残差模型可以借鉴：

```text
seasonality block + residual block
```

---

## 5. DLinear / NLinear

非常简单，但在很多长序列预测任务中表现稳定。可以作为低复杂度深度模型 baseline。

---

## 6. Decision-focused Learning

这是最契合本赛题的思想。

传统做法：

```text
预测价格最准确 -> 再优化策略
```

更好的做法：

```text
预测结果服务于最终收益最大化
```

对应到本项目，就是：

* 不只看 RMSE；
* 用窗口收益、regret、capture ratio 选择模型；
* 训练窗口 ranking 模型；
* 用收益回测 early stopping。

---

# 十二、建议你马上改的代码方向

## 1. 增加 oracle 最优收益评估

在 `validate_profit.py` 里增加：

```python
def oracle_profit_one_day(true_prices):
    result = optimize_one_day(true_prices, threshold=0.0)
    return result.best_profit if result.traded else 0.0
```

然后每天计算：

```python
oracle_profit
capture_ratio = profit / oracle_profit if oracle_profit > 0 else 0
regret = oracle_profit - profit
```

这样你才能知道自己离理论上限差多少。

---

## 2. 增加窗口命中分析

每天记录：

```text
oracle_charge_start
oracle_discharge_start
pred_charge_start
pred_discharge_start
abs_charge_error
abs_discharge_error
```

如果你发现：

```text
charge_start 基本稳定，但 discharge_start 经常错
```

那就重点优化高价尖峰预测。

如果发现：

```text
discharge_start 对，但 charge_start 错
```

那就重点优化低价/新能源过剩时段。

---

## 3. 增加价格滞后特征

建议新增一个函数：

```python
def add_price_history_features(feature_df, label_df):
    # label_df: times, A
    # feature_df: times, boundary features
    # 构造同 slot 历史价格统计
```

重点不要只做均值，而要做：

```text
same_day_last_year
month_slot_mean
month_slot_median
month_slot_q25
month_slot_q75
dow_slot_mean
slot_recent_mean
slot_recent_std
```

---

## 4. 修改 NWP 读取为自适应维度

不要写死：

```python
data[hour, channel, :, :]
```

建议根据维度长度判断：

```python
if data0.shape[0] == len(channels):
    # channel x hour x lat x lon
    arr = data0[channel_idx, hour, :, :]
else:
    # hour x channel x lat x lon
    arr = data0[hour, channel_idx, :, :]
```

更严谨的做法是读取 `.nc` 维度名，但你现在用 h5py，可能需要先打印文件结构。

---

## 5. 训练残差而不是原始价格

把训练目标从：

```python
y = A
```

改成：

```python
seasonal_prior = hist_month_slot_mean 或 hist_dayofyear_slot_mean
y = A - seasonal_prior
```

预测后：

```python
pred_price = seasonal_prior + pred_residual
```

这个通常比直接预测电价更稳。

# 十五、最终建议的冲榜路线

我建议你不要继续盲目加模型，而是按这个顺序冲分：

```text
1. 修正提交脚本，确保验证和提交一致
2. 修正 NWP 维度与 UTC/BJ 时间
3. 建立 oracle/regret/capture ratio 诊断
4. 增加价格历史和同年/去年同日先验
5. 改成 seasonal_prior + residual 模型
6. 加月度 threshold 和保守交易规则
7. 做窗口 ranking 模型
8. 做交易分类模型
9. 最后再加深度模型集成
```

你当前仓库已经完成了 60% 的工程基础，但离高分还差最关键的 40%：

> **从“预测电价”升级为“预测可套利窗口 + 控制亏损风险”。**

最优提分方向不是“再换一个更复杂模型”，而是把整个项目改成：

```text
价格预测模型
+ 季节先验模型
+ 窗口收益排序模型
+ 是否交易分类模型
+ 风险阈值控制
+ 多折收益回测
```

这样才真正对齐比赛评分。
